Generation Pipeline — Architectural Approaches

How the PipelineCoordinator orchestrates multi-pass generation across agent families

A

Sequential Pipeline with Refinement Loops

Each pass runs in strict order. After all passes complete, a validation pass checks constraints (connectivity, difficulty curve, loot budget). If validation fails, specific passes re-run with adjusted params. Simple to reason about, easy to debug.

① TerrainAgent → noise maps, biome grid
② WaterAgent → rivers, lakes
③ CaveCarverAgent → carve caves/tunnels from noise
④ StructureAgent → place rooms using topology graph
⑤ ConnectorAgent → corridors, doors, stairs
⑥ RoomPurposeAgent → assign themes, roles
⑦ EncounterAgent → place creatures per budget
⑧ TrapAgent → place traps per danger map
⑨ LootAgent → distribute treasure per budget
⑩ DressingAgent → furniture, decor, atmosphere
⑪ ValidationPass → check constraints, re-run ④-⑩ if needed
⑫ RendererAgent + Unity Exporters

Pros

  • Simple mental model
  • Easy to debug — each pass is isolated
  • Deterministic ordering

Cons

  • Late validation means wasted work
  • Refinement loops can oscillate
  • Hard to express cross-cutting constraints
B

Two-Phase: Macro Layout → Micro Design

Split generation into two distinct phases. Phase 1 (Macro) creates the spatial skeleton: terrain, caves, room positions, connectivity graph, zone boundaries. Phase 2 (Micro) fills each room/zone independently: purpose, encounters, traps, loot, dressing. Macro validates once; micro runs per-room in parallel.

PHASE 1 — MACRO (spatial skeleton)
① TerrainAgent → noise maps, biome grid
② WaterAgent → rivers, lakes
③ CaveCarverAgent → carve open spaces from noise
④ TopologyAgent → build room graph (linear/loop/hub)
⑤ StructureAgent → place rooms to satisfy graph
⑥ ConnectorAgent → corridors, doors, transitions
── Macro Validation ──

PHASE 2 — MICRO (per-room design)
⑦ RoomPurposeAgent → assign role from profile
⑧ EncounterAgent → creatures per budget + pacing
⑨ TrapAgent → traps per danger map
⑩ LootAgent → treasure per risk-reward
⑪ DressingAgent → furniture, atmosphere
── Budget Validation ──

⑫ RendererAgent + Unity Exporters

Pros

  • Clean separation of spatial vs content concerns
  • Macro validates early before micro work begins
  • Micro pass is embarrassingly parallel (per-room)
  • Easy to swap room design without touching layout

Cons

  • Slightly more complex coordinator
  • Cross-room constraints (e.g. "exactly one boss room") need phase boundary logic
C

Three-Phase: Terrain → Layout → Population

Like B but with an explicit middle phase for layout intelligence. Phase 1 (Terrain) is pure noise/geology. Phase 2 (Layout) handles topology, rooms, connections — the architectural design. Phase 3 (Population) fills rooms with encounters, loot, traps, dressing. Each phase validates independently.

PHASE 1 — TERRAIN (geology)
① TerrainAgent → elevation, moisture, biome
② WaterAgent → hydrology
③ CaveCarverAgent → natural cavities
── Terrain Validation ──

PHASE 2 — LAYOUT (architecture)
④ TopologyAgent → room graph + zone boundaries
⑤ StructureAgent → place rooms into terrain
⑥ ConnectorAgent → corridors, doors, stairs
⑦ PathfindingAgent → verify connectivity
── Layout Validation ──

PHASE 3 — POPULATION (content)
⑧ RoomPurposeAgent → assign roles from profile
⑨ EncounterAgent → creatures per budget + pacing
⑩ TrapAgent → traps per danger map
⑪ LootAgent → treasure per risk-reward
⑫ DressingAgent → furniture, atmosphere
⑬ LabelingAgent → names, descriptions
── Population Validation ──

⑭ RendererAgent + SpawnAgent + Unity Exporters

Pros

  • Cleanest separation — each phase has one job
  • Three validation gates catch errors early
  • Terrain phase is reusable (same terrain, different layouts)
  • Layout phase can retry without regenerating terrain
  • Population is fully independent of geometry

Cons

  • Most complex coordinator
  • More validation passes to implement

⭐ Recommendation: Option C

Three phases maps perfectly to the design decisions we've made: noise terrain (Phase 1) feeds topology selection and room placement (Phase 2) which feeds D&D-style encounter/loot/trap design (Phase 3). Each phase validates independently, and the coordinator can retry Phase 2 without burning Phase 1 work. The "extra complexity" is just two if-statements in the coordinator.
