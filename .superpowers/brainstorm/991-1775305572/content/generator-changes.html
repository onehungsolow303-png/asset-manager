Section 4: Map Generator Changes + Unity Export

What changes in the existing codebase to support z-levels

SharedState Changes
BEFORE (flat):
SharedState:
  elevation: float32[H,W]
  terrain_color: uint8[H,W,3]
  walkability: bool[H,W]
  water_mask: bool[H,W]
  structure_mask: bool[H,W]
  entities: list[Entity]
  paths: list[PathSegment]
  labels: list[Label]
AFTER (layered):
SharedState:
  config: MapConfig
  ground: ZLevel          # z=0, always exists
  levels: dict[int, ZLevel]  # all z-levels
  transitions: list[Transition]
  spawns: list[SpawnPoint]
  metadata: dict

  # Backwards compat properties:
  @property elevation → ground.elevation
  @property terrain_color → ground.terrain_color
  @property walkability → ground.walkability
  ... etc
New Data Classes
@dataclass
class ZLevel:
  z: int
  terrain_color: uint8[H,W,3]
  walkability: bool[H,W]
  structure_mask: bool[H,W]
  water_mask: bool[H,W]
  entities: list[Entity]
  labels: list[Label]

@dataclass
class Transition:
  x, y: int          # position on map
  from_z: int         # source z-level
  to_z: int           # destination z-level
  type: str           # "stairs_up", "stairs_down",
                      # "ladder", "trapdoor", "entrance"

@dataclass
class SpawnPoint:
  x, y: int
  z: int
  token_type: str     # "player", "enemy", "npc"
  stats: dict         # HP, AC, STR, DEX, etc.
  ai_behavior: str    # "patrol", "guard", "chase"
  name: str           # "Goblin Grunt", "Blacksmith"
Agent Changes Summary
Agent Change Effort
shared_state.py Add ZLevel, Transition, SpawnPoint classes. Wrap existing arrays in ground ZLevel. Add backwards-compat properties so existing agents still work unchanged. Medium
structure_agent.py Multi-floor generation for buildings. Cottages: ground + roof. Castles: basement + ground + upper + battlements. Dungeons: entrance + N sub-levels. Creates ZLevels and Transitions (stairs). Large
terrain_agent.py No change for z=0. Sub-level terrain (cave floors, stone floors) generated by structure_agent when it creates deeper levels. None
asset_agent.py Place assets on the correct z-level. Outdoor assets on z=0, indoor furniture on the building's floor level. Small
renderer_agent.py Generate per-layer PNGs instead of single composite. Also still produces a flattened "preview" PNG for the GUI. Medium
NEW: spawn_agent.py New agent that places player spawn, enemies, and NPCs with stat blocks. Enemies scale by map type/difficulty. Placed after structures. Medium
Unity exporters Export z-level data, transitions, and spawn points to JSON. Unity scripts get layer visibility toggle + spawn loading. Medium
Layered Output Format
output/village_forest_42_20260404_065500/
  preview.png                 # Flattened preview (for GUI)
  map_data.json               # Full layered data for viewer
    ├─ config (size, biome, seed, etc.)
    ├─ z_levels: [
    │    { z: -1, terrain: "z_neg1.png", walkability: [...], entities: [...] },
    │    { z: 0,  terrain: "z_0.png",    walkability: [...], entities: [...] },
    │    { z: 1,  terrain: "z_1.png",    walkability: [...], entities: [...] },
    │  ]
    ├─ transitions: [ {x, y, from_z, to_z, type} ... ]
    ├─ spawns: [ {x, y, z, type, stats, name} ... ]
    └─ labels: [ {x, y, z, text, category} ... ]
  z_neg1.png                  # Terrain image for z=-1
  z_0.png                     # Terrain image for z=0 (ground)
  z_1.png                     # Terrain image for z=1 (roofs/upper)
  unity_export/               # Unity files (enhanced with layers)

Backwards Compatibility

Existing agents (terrain, water, pathfinding, labeling) work unchanged — they write to shared_state.terrain_color which maps to ground.terrain_color via property. Only structure_agent and asset_agent need awareness of z-levels. A new spawn_agent handles creature placement. The renderer produces both per-layer PNGs and a flattened preview.
