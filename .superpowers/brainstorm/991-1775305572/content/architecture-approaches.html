Architecture: How to build the layered renderer + playtest engine

Three approaches with different complexity/capability trade-offs

A

Pygame Viewer (Recommended)

Build the playtest viewer as a pygame application. Pygame gives us a real game loop with keyboard input, sprite rendering, per-pixel drawing, and 60fps updates. The map generator stays as-is (generates layered data), and a new viewer/ module renders it interactively.

Pros

  • Real game loop -- smooth movement, real-time input
  • Hardware-accelerated rendering
  • Natural fit for fog of war, LOS, combat animations
  • Huge ecosystem of tutorials and game dev patterns
  • Can later become an actual game prototype

Cons

  • New dependency (pygame)
  • Separate window from the GUI
  • More code (~1500-2000 lines for full playtest)
B

Tkinter Canvas Viewer

Build the viewer directly into the existing tkinter GUI. Use a Canvas widget with tile-based rendering. The playtest mode is a tab/panel within the current app.

Pros

  • No new dependencies
  • Integrated with existing GUI
  • Single window experience

Cons

  • Tkinter Canvas is slow for real-time games
  • No proper game loop -- hacked with after() calls
  • Poor keyboard handling for WASD movement
  • Fog of war / LOS very hard to do performantly
  • Will feel sluggish compared to real game engines
C

Web-Based Viewer (HTML5 Canvas)

Build the viewer as a local web app. The Python backend serves map data via a local HTTP server, and a JavaScript frontend renders tiles on an HTML5 Canvas with WebGL acceleration.

Pros

  • Excellent rendering performance (Canvas/WebGL)
  • Cross-platform UI with modern look
  • Easy to add UI controls (HTML buttons, sliders)
  • Could become a web deployment later

Cons

  • Two-language system (Python + JavaScript)
  • Server/client complexity for communication
  • Browser has to be open separately
  • Debugging split across two runtimes

My Recommendation: A (Pygame)

For a playtest viewer with movement, combat, fog of war, and LOS -- you need a real game loop. Pygame is the only option here that won't fight you at every step. Tkinter will feel sluggish and hacky for real-time input. The web option works but splits your codebase across two languages.

Pygame is a single pip install pygame, gives you everything a 2D game engine needs, and the viewer can evolve into an actual game prototype if you want.
